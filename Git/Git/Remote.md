**اشتباه رایج: هر بار نیاز نیست!**

​وقتی پروژه‌ای را از گیت‌هاب [[۴. کپی برداری (Clone)]] می‌کنی، گیت خودش به‌صورت خودکار آدرس آن را به اسم [[origin]] ذخیره می‌کند. یعنی تو اصلاً لازم نیست کاری انجام بدهی. فقط می‌زنی git [[Git push]] و گیت می‌داند مقصد کجاست.
​اما اگر پروژه را در سیستم خودت شروع کرده باشی ([[Git init]])، گیت اصلاً نمی‌داند گیت‌هاب چیست! اینجاست که فقط یک‌بار این دستور را می‌زنی:
```
git remote add origin https://github.com/user/repo.git
```
از این لحظه به بعد، گیت می‌فهمد که کلمه «origin» یعنی آن آدرسِ طولانی گیت‌هاب.


___

​**۲. چرا اسمش را گذاشتیم origin؟**

​این یک قرارداد است. تو می‌توانی اسمش را بگذاری my-server یا gholi! اما همه در دنیا از origin استفاده می‌کنند. وقتی می‌گویی:
```
git push origin main
```
داری به گیت می‌گویی: «تغییرات من را بفرست به آدرسی که با نام origin ذخیره کردم، توی شاخه ([[git branch]]) اصلی.»

___

**سناریویی که ممکن است گیجت کند:**

​بسیاری از حرفه‌ای‌ها از ==چند ریموت== هم‌زمان استفاده می‌کنند. اینجاست که قدرت git remote معلوم می‌شود.
​تصور کن:
- ​یک آدرس داری برای گیت‌هاب (اسمش را می‌گذاری origin).
- ​یک آدرس داری برای سرور شرکت (اسمش را می‌گذاری office).
- ​حالا وقتی بخواهی کد را برای همکارانت بفرستی می‌زنی:
```
git push origin main
```
- و وقتی بخواهی همان کد را روی سرور شرکت بفرستی تا سایت آپدیت شود، می‌زنی:
```
git push office main
```



___

**خلاصه‌ی کلام:**
​Git Remote:
تعریف کردنِ رابطه (لینک) بین فولدرِ روی کامپیوترت و مخزنِ روی اینترنت است.
​کی استفاده می‌کنیم؟

**==وقتی می‌خواهیم به گیت بفهمانیم "مقصد" کجاست.==**

​تعداد دفعات: معمولاً فقط یک‌بار برای هر پروژه تنظیم می‌شود.

___
می‌خواهی لیست ریموت‌هایی که همین الان روی پروژه‌ات تنظیم شده را ببینی؟ کافی است در ترمینالِ آن پروژه تایپ کنی:
```
git remote -v
```
